# Internationalization (i18n)

## Overview

Synedria is built with multilingual support from day one. The i18n infrastructure is set up during initial development even though the platform launches with only Italian and English. This is a structural decision: retrofitting i18n across all pages later is significantly more expensive than establishing the pattern from the start.

The target is Italian territory, but the tech audience often prefers English interfaces. Both languages are available from launch. All UI strings are externalized — no hardcoded text in components.

## User Stories

1. As an Italian user, I want to use the platform in Italian so I can navigate comfortably in my language.
2. As a tech-savvy user, I want to switch to English so I can use the interface in the language I'm most comfortable with for technical content.
3. As a user, I want the platform to detect my browser language and show the appropriate version automatically.
4. As a user, I want to manually switch language at any time and have my preference remembered.
5. As a search engine, I want to see proper `hreflang` tags so I can index the correct language version for each market.

## Functional Requirements

### Locale Routing

- FR-1: Locales are embedded in the URL path: `/it/...` for Italian, `/en/...` for English.
- FR-2: The default locale is Italian (`it`).
- FR-3: Supported locales in phase 1: `it` (Italian), `en` (English).
- FR-4: Requests to the root `/` are redirected to the appropriate locale based on browser `Accept-Language` header.
- FR-5: If the browser language doesn't match any supported locale, default to Italian.

### Language Switcher

- FR-6: A language switcher component is available in the site header/navigation.
- FR-7: Switching language navigates to the same page in the other locale (e.g., `/it/groups/devops-milano` → `/en/groups/devops-milano`).
- FR-8: Language preference is persisted (cookie or localStorage) and respected on subsequent visits.

### Translation System

- FR-9: All UI strings are stored in translation files, one per locale (e.g., `messages/it.json`, `messages/en.json`).
- FR-10: No hardcoded text in components — all visible text uses translation keys.
- FR-11: Translation keys are organized by feature/page (e.g., `auth.login`, `groups.create`, `search.noResults`).
- FR-12: Support for interpolation (dynamic values in strings) and pluralization.

### SEO

- FR-13: Each page includes `hreflang` tags pointing to all available locale versions.
- FR-14: `<html lang="...">` attribute is set correctly based on the current locale.
- FR-15: SEO metadata (title, description, Open Graph) is translated per locale.
- FR-16: The sitemap includes all locale variants of each page.

### User-Generated Content

- FR-17: User-generated content (group descriptions, objectives, messages) is NOT translated — it stays in the language the user wrote it.
- FR-18: The UI around user-generated content is translated; the content itself is not.

## Technical Notes

- **Next.js App Router i18n**: Use the `[locale]` dynamic segment pattern in the app directory: `app/[locale]/layout.tsx`, `app/[locale]/groups/[slug]/page.tsx`, etc.
- **Middleware**: `middleware.ts` handles locale detection from `Accept-Language` header and redirects from `/` to `/[locale]/`.
- **Library**: Use `next-intl` — lightweight, well-maintained, designed for App Router. Provides `useTranslations` hook for Client Components and `getTranslations` for Server Components.
- **Translation files**: JSON format in a `messages/` directory at project root. Structure: flat or nested keys (e.g., `{"auth": {"login": "Accedi", "logout": "Esci"}}`).
- **Type safety**: `next-intl` supports TypeScript — translation keys are type-checked, preventing typos and missing translations.
- **Fallback**: If a key is missing in the current locale, fall back to Italian (the primary locale). Log a warning in development.
- **Static generation**: Pages using i18n can still benefit from static generation. `generateStaticParams` should emit all supported locales.

## Dependencies

### Depends on

- Next.js App Router (project scaffolding)
- `next-intl` package (or equivalent)

### Depended on by

- **All features**: Every page and component uses the i18n system for UI strings
- **05-Group Page**: locale routing affects group page URLs; hreflang tags needed
- **06-Search/Discovery**: search page URL includes locale; UI strings translated

## Out of Scope

- Translation of user-generated content (group descriptions, messages)
- Automatic translation (Google Translate integration, etc.)
- RTL language support (no RTL languages planned)
- More than 2 locales in phase 1
- Translation management platform (Crowdin, Lokalise — overkill for 2 locales)
- Content localization (adapting content for cultural context beyond translation)

## Open Questions

1. **Locale in URL vs subdomain**: URL path (`/it/`, `/en/`) is the standard for Next.js App Router and simpler to implement. Subdomains (`it.synedria.app`) are an alternative but add DNS/deployment complexity. Decision: URL path.
2. **Translation workflow**: Who translates? For 2 locales, manual translation in JSON files is sufficient. No tooling needed yet.
3. **Locale-specific content**: Should the landing page have different content per locale (not just translated), or identical content in both languages? Start with identical translated content; customize later if needed.
